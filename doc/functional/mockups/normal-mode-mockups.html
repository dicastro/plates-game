PLATES — Functional Specification

Normal Mode — Game Screen

Visual and behavioral reference for the Normal Mode game screen (NORMAL_GAME). Covers screen composition, states, overlays, and the responsive/rotation rules that apply to all of them. Theme: cantabria-verde. Proportions and spacing shown are illustrative; exact pixel values are an implementation detail.

1. Collapsed state

The keyboard is hidden by default. A single CTA occupies the space where the keyboard will appear. Tapping the CTA expands the input row and keyboard (section 2). The best-score panel is tappable and opens the attempts detail overlay (section 5). A Share icon — icon only, no label — sits next to the panel whenever a valid attempt exists for the current day; it is absent otherwise.

2. Expanded state

The plate and best-score panel stay visible above the input row while the keyboard is expanded. The keyboard is anchored to the bottom edge of the screen, so it sits where a player's thumbs naturally rest on a handheld device. Backspace is part of the keyboard's bottom row, next to Z, matching common mobile keyboard layouts. The input row contains only Submit (left edge) and Close (right edge), kept apart from each other to avoid accidental taps. Submit is enabled only once the typed word contains the plate's three consonants in order.

3. Landscape

When enough vertical height is available, the bonus/difficulty badges sit under the plate — the same composition as the collapsed state — and the keyboard spans the full width, anchored to the bottom edge of the screen. The best-score panel keeps its attempt dots. When vertical height is limited, the keyboard occupies a side panel instead: the plate, badges, and best-score panel form a column on the left, and the entire left-column-plus-keyboard block is centered vertically as a single unit within the screen — the two halves are aligned to each other's top edge, not independently centered.

4. Submission flow overlays

Submitting a word collapses the keyboard, shows the CTA in its collapsed position, and displays a loading overlay while the Worker validates the word against the dictionary. The Worker's response opens one of three result overlays. The "new best" variant offers sharing the word and score just achieved. The "valid, not best" variant offers sharing today's existing best score instead — framed as challenging a friend — since the word just submitted did not change that best score. The "not in the dictionary" variant offers no sharing option, since the attempt produced no score. Dismissing any result overlay returns to the collapsed state with the CTA.

6. Attempts exhausted

Once the daily attempt limit is reached, the CTA becomes disabled and shows a countdown to the next plate instead of its usual label. The countdown is a local, non-authoritative display; it is informational only, not an anti-cheat mechanism. Below it, a secondary, independent button offers an extra attempt in exchange for watching a rewarded ad. This button is gated by the player attribute adsEnabled: boolean on PlayerProfile (default true): when false, the button does not render at all. Granting the extra attempt always requires Worker-side confirmation that the rewarded ad actually completed — it is never a client-only unlock.

7. Responsive degradation hierarchy & device rotation

No element on this screen has a fixed pixel height; every size derives from the available width and height of its container, so the same rules apply across portrait, landscape, and rotation in either direction.

Visibility priority, highest to lowest:
1. The plate — always visible. It is the game's identity and the player's only reference while typing.
2. The keyboard and input row — always visible whenever expanded.
3. Bonus/difficulty badges — the first element hidden if vertical space is insufficient.
4. The best-score panel, including its attempt dots — hidden only if hiding the badges was not enough.

Implementation approach:
• Each keyboard key uses aspect-ratio: 1 / 1 (or close to it) instead of a fixed height — width comes from the grid (repeat(N, 1fr)) over the available width, and height derives from that width automatically.
• The badge-then-best-score-panel hide order is driven by CSS Container Queries (@container) on a sized wrapper around the game area, not by @media viewport queries. Exact height thresholds are an implementation detail.
• No JavaScript measures or assigns pixel sizes anywhere in this scheme; sizing is declarative CSS end to end.

Device rotation: game state — the word being typed, attempts used, best score — lives in React state, never in the DOM or derived from layout. Rotating the device only changes the container size that the CSS above reacts to. No component inside the game screen may be keyed by orientation or unmounted/remounted on resize, so a rotation never resets or loses in-progress state.
